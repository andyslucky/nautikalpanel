title: Managing Servers
description: Starting, stopping, and managing game servers.
---

Once you've created a server, you'll need to manage its lifecycle. This guide covers all management operations.

## Starting a Server

1. Locate the server in the dashboard
2. Ensure status shows "Offline"
3. Click **Start** button
4. Status transitions: Offline → Pending → Running

The status updates in real-time via WebSocket, so you'll see the transition happen immediately without refreshing the page.

### What Happens During Start

1. Kubernetes pod is created from the pod template
2. Container image is pulled (if not cached)
3. Game server process starts
4. Service becomes available with external IP
5. SFTP credentials are generated and stored in K8s secret
6. Dashboard receives real-time updates as the pod transitions through states

### Troubleshooting Startup

If the server stays in "Pending":
- Check Kubernetes events: `kubectl describe pod <pod-name> -n nautikal`
- Verify image is valid and accessible
- Check resource requests don't exceed node capacity

## Stopping a Server

1. Click **Stop** on a running server
2. Status transitions: Running → (brief Pending) → Offline

### What Happens During Stop

1. Pod is deleted
2. SFTP credentials secret is deleted
3. PVC and data are preserved
4. Service remains (for LoadBalancer IP retention)

## SFTP Access Mode

For file management without running the game:

1. Ensure server is Offline
2. Click **SFTP Only**
3. A pod with only SFTP container starts
4. Access files via SFTP client

### Why Use SFTP Only?

Use SFTP Only mode for lower resource usage, to upload worlds or mods before the first start, to backup or restore data, or when the game server doesn't need to be running.

## Editing Server Settings

:::note
Servers can only be edited when offline.
:::

1. Stop the server if running
2. Click **Edit** on the server card
3. Modify desired settings
4. Click **Save**

### What Can Be Changed

Server name, game version, max players, icon URL, description, container image, resources (CPU/memory), environment variables, command, and pod template can all be edited. Game type, storage size, service type, and ports cannot be changed.

### Why Some Fields Are Not Editable

Storage and network settings are tied to Kubernetes resources created during server initialization. Changing these would require:
- Deleting and recreating PVCs (data loss risk)
- Recreating Services (IP address changes)

## Deleting a Server

1. Click **Delete** on any server card
2. Confirm the deletion

:::danger
Deletion is permanent and removes:
- Server configuration from database
- Persistent Volume Claim (all data)
- Service (releases IP address)
- Any running pods
:::

### Before Deleting

- Download important files via SFTP
- Note your configuration if you might recreate

## Monitoring Server Status

### Dashboard Status

The colored status dot indicates running (green) when the pod is active and server should be accessible, offline (black) when no pod is running, and pending (yellow) when the pod is being created or terminated.

### Detailed Status

Hover over or check the pod status for more details, including pending when waiting for resources, running for an active container, failed when the container exited with an error, and unknown for API communication issues.

## Refresh Behavior

The dashboard automatically refreshes at the configured interval (default: 30 seconds, configurable in Settings) to update status for all servers.

## Bulk Operations

Currently, operations must be performed on individual servers. Future versions may include bulk start/stop, bulk delete, and filtering and search capabilities.

## Resource Cleanup

When a server is deleted, Nautikal Panel deletes the pod if running, deletes the PVC and all data, deletes the Service, deletes any SFTP credentials secrets, and removes the record from the database. No orphaned resources should remain.
