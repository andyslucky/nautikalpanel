Creating a game server in Nautikal Panel is straightforward thanks to the template system. This guide covers all the options available.

## Opening the Create Dialog

1. Click **+ Add Server** on the dashboard
2. The Create Server modal opens with multiple tabs

## General Tab

Basic server identification and settings.

### Server Name

A friendly name to identify this server, shown in the dashboard and logs, which can be changed later via Edit.

### Template Selection

Select from available templates in the dropdown. Templates define default configuration for a game type and populate all other fields, which you can modify after selection.

### Game Version

Optional field for tracking that is not used by Nautikal Panel directly but is useful for documentation.

### Max Players

Optional player limit that some games use for configuration and is informational for others.

### Game Type

Read-only field derived from the template, used for Kubernetes resource naming.

### Icon URL

Optional URL for a server icon displayed in the dashboard, which can be any accessible image URL.

### Description

Optional server description for your reference.

## Pod Config Tab

Container and resource configuration.

### Container Image

The Docker image to run, such as `itzg/minecraft-server` for Minecraft, `lloesche/valheim-server` for Valheim, or any valid Docker image reference.

### CPU Resources

Dual sliders for CPU allocation control the guaranteed CPU (request) in millicores and the maximum CPU allowed (limit), with example values including `250m` for 0.25 cores, `1000m` for 1 core, and `2000m` for 2 cores.

### Memory Resources

Dual sliders for memory control the guaranteed memory (request) and maximum memory allowed (limit), with example values including `256Mi` for 256 megabytes, `1Gi` for 1 gigabyte, and `4Gi` for 4 gigabytes.

### Command

Custom command to run in the container. Enter comma-separated values like `java, -Xms1G, -Xmx4G, -jar, server.jar`, or leave empty to use the image default.

### Environment Variables

Key-value pairs passed to the container, such as `EULA` set to `TRUE`, `VERSION` set to `1.20.4`, or `DIFFICULTY` set to `normal`. Click **Add Env Var +** to add new entries.

## Storage Config Tab

Persistent storage configuration.

### Storage Size

Amount of disk space allocated as a number with unit (Mi or Gi). This cannot be changed after creation, so choose based on game requirements.

### Size Unit

Storage size unit, either Mi for megabytes or Gi for gigabytes.

### Container Path

Where storage is mounted in the container. Common paths include `/data`, `/server`, and `/world`, which must match what the game expects. The template provides a sensible default.

### Storage Class

Optional Kubernetes storage class. Leave empty for the cluster default, or specify for special storage needs like SSD or NFS.

## Service Config Tab

Network and service exposure.

### Service Type

Choose LoadBalancer for cloud environments with external load balancers, NodePort for bare metal to expose on node IP, or ClusterIP for internal-only access requiring port-forward.

### IP Address

Optional specific IP for LoadBalancer. Leave empty for automatic assignment, or specify for static IP requirements.

### Ports

Ports to expose, including the external port number and protocol (TCP or UDP). Click **Add Port +** for multiple ports.

## Misc Tab

Advanced template overrides.

:::warning
These options are for advanced users. Incorrect values can cause failures.
:::

### Init Template

Path to Jinja template for initial resources, which creates the PVC and Service. The default is `default/init.yaml.jinja`, which you can override for custom resource creation.

### Pod Template

Path to Jinja template for the pod, which creates the game server pod. The default is `default/pod_template.yaml.jinja`, which you can override for custom pod configuration.

## Creating the Server

Click **Create** to:
1. Save the server configuration to the database
2. Create initial Kubernetes resources (PVC, Service)
3. Return to dashboard with server showing "Offline" status

The server is now ready to be started.

## Editing After Creation

Servers can only be edited when offline:
1. Stop the server if running
2. Click **Edit** on the server card
3. Modify available fields (not all are editable)
4. Click **Save**

### Editable Fields

Server name, game version, max players, icon URL, description, pod configuration (image, resources, env, command), and pod template path can all be modified.

### Non-Editable Fields

Game type, storage configuration, service configuration, and init template are set at creation and cannot be changed.
